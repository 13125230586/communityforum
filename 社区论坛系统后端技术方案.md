# 社区论坛系统后端技术方案

## 一 项目概述

### 1.1 项目目标

本项目目标是从 0 到 1 搭建一个单体架构的社区论坛系统后端 为后续前端首页 帖子详情 发布编辑帖子页面 个人中心 后台管理系统提供统一稳定的接口服务

一期核心范围包括：

- 用户管理
- 板块分类管理
- 帖子管理
- 评论管理
- 系统设置
- 系统统计
- OSS 图片上传

### 1.2 当前技术决策

本项目不机械照搬商城项目 而是只复用适合论坛场景的技术和结构

最终后端技术栈为：

- Spring Boot 2.7.6
- Spring MVC
- MyBatis Plus
- MySQL 8
- Spring AOP
- Knife4j
- HttpSession
- 阿里云 OSS

明确不使用：

- Redis
- Spring Session Redis
- RabbitMQ
- XML Mapper

### 1.3 设计原则

- 保持单体架构 简化本地开发和调试
- 按业务域拆分包结构 避免代码平铺
- 统一接口返回结构和异常处理
- 尽量复用 MyBatis Plus 标准能力 不引入多余复杂度
- 列表查询和详情查询分离 大字段拆表
- 优先保证论坛主链路能快速落地

### 1.4 当前实现状态

截至当前工程版本 一期核心后端已经补齐到可联调状态 主要包括：

## 三 总体架构设计

### 3.1 架构模式

采用单体项目 + 按业务域模块化分包

整体分层如下：

- controller：接口层
- service：业务服务层
- mapper：数据访问层
- entity：数据库实体
- model/dto：请求模型
- model/vo：响应模型
- common：统一响应和基础请求
- config：系统配置
- aop：权限切面
- manager：第三方能力封装
- utils：工具类

### 3.2 为什么采用业务域模块化分包

如果继续沿用传统平铺目录 后续帖子 评论 举报 审核 系统设置 统计代码会快速膨胀

论坛比商城更偏内容治理和互动关系 采用业务域拆分后：

- 用户逻辑更集中
- 板块和帖子边界更清晰
- 评论树设计更容易独立演进
- 系统配置和统计不会污染核心业务模块

### 3.3 当前推荐目录结构

```text
├── pom.xml

    │   ├── CommunityforumBackendApplication.java
    │   ├── annotation
    │   ├── aop
    │   ├── common
    │   ├── config
    │   ├── constant
    │   ├── controller
    │   ├── exception

    │       ├── user

    │       ├── system
    │       ├── statistics

    └── resources
        ├── application.yml
        └── application-local.yml
```

## 四 技术栈说明

### 4.1 Spring Boot

用途：

- 提供 Web 容器和自动装配能力
- 承载 REST API
- 统一项目启动与配置管理

版本：

- 2.7.6

### 4.2 MyBatis Plus

用途：

- 负责基础 CRUD
- 提供 `BaseMapper` 和 `ServiceImpl`
- 提供分页插件
- 提供逻辑删除能力

约束：

- 不使用 XML Mapper
- 不配置 `mapper-locations`
- 常规查询走 `QueryWrapper`
- 后续复杂统计再按实际情况补充自定义 SQL

### 4.3 MySQL

用途：

- 存储用户 板块 帖子 评论 举报 配置 统计等核心业务数据

要求：

- 字符集统一 `utf8mb4`
- 统一 InnoDB
- 表和字段必须有 `COMMENT`
- 所有核心查询场景必须建索引

### 4.4 Spring AOP

用途：

- 统一权限校验
- 后续可扩展操作日志审计

当前实现：

- `@AuthCheck`
- `AuthInterceptor`
- `@OperationLog`
- `OperationLogAspect`

### 4.5 HttpSession

用途：

- 保存登录态
- 适合当前单机本地开发环境

说明：

- 当前不引入 Redis 会话共享
- 服务重启后会话失效 这在开发环境可接受

### 4.6 Knife4j

用途：

- 提供接口文档
- 便于前后端联调

### 4.7 阿里云 OSS

用途：

- 上传用户头像
- 上传板块图标
- 上传帖子封面
- 上传帖子正文图片

当前策略：

- 一期已支持上传返回 URL
- 已补充文件记录表和后台文件管理
- 业务表直接存储 OSS URL

## 七 登录和权限设计

### 7.1 登录流程

登录步骤：

- 用户提交账号密码
- 后端校验参数
- 对密码进行加盐 MD5
- 查询数据库匹配用户
- 校验账号状态
- 更新最后登录时间和最后登录 IP
- 将用户实体写入 `HttpSession`
- 返回脱敏后的登录用户信息

### 7.2 登录态保存

登录态 key：

- `user_login`

当前存储方式：

- `HttpSession`

工具类：

- `utils/SessionUtil.java`

### 7.3 权限注解

权限控制采用：

- `@AuthCheck`

切面：

- `aop/AuthInterceptor.java`

处理逻辑：

- 无角色要求时 只校验是否登录
- 有角色要求时 判断当前用户角色是否满足
- 管理员默认可覆盖普通角色权限

### 7.4 角色设计

建议角色值：

- `user`
- `moderator`
- `admin`

一期主要使用：

- `user`
- `admin`

`moderator` 先预留 后续可扩展版主能力

## 十三 开发规范约束

### 13.1 接口规范

按当前规则执行：

- GET 负责查询
- POST 负责新增 修改 删除
- POST 请求统一 DTO
- VO 负责返回

### 13.2 日志规范

日志要求：

- 中文简短描述
- 关键 ID 放前
- 变量使用 `uid:{}` `postId:{}` 这种格式
- 对象日志使用 `JSON.toJSONString`

### 13.3 字段命名规范

避免使用：

- `status`
- `isDeleted`
- `isAnonymous`

当前统一风格：

- `userStatus`
- `boardStatus`
- `postStatus`
- `commentStatus`
- `auditStatus`
- `anonymousFlag`
- `deleteFlag`

### 13.4 MyBatis Plus 规范

- 优先使用 `ServiceImpl`
- 优先使用 `QueryWrapper`
- 不依赖 XML
- 分页统一 `Page<>`
